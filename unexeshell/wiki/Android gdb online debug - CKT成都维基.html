GDB简介[编辑 | 编辑源代码]

关于GDB的简介谷哥和度娘搜索就知道，一句话概括就是：GDB 是UNIX，GNU下最牛逼的调试工具。

既然GDB这么牛逼， Android 这个集广大优秀软件于一体的系统当然也会集成进来，后面会看到

Android 为GDB 的易用性做了封装工作，那就是gdbclient脚本命令，gdbserver attach process工具。

GDB 一般用于debug Native code 和 kernel，本wiki主要讲解Android 环境online debug Native code

至于kernel 部分的panic分析，前面wiki也有讲解到，可以参考如下wiki：

http://10.120.10.100:9002/Gdb_objdump_-s

http://10.120.10.100:9002/MTK_kernel_vmcore

http://10.120.10.100:9002/GDB_vmlinux_vmcore

gdbserver and gdbclient[编辑 | 编辑源代码]

在线调试GDB，一般都需要gdbserver 和 gdbclient 来交互， gdbserver 运行在目标板子（这里就是我们的手机），gdbclient一般是交叉工具链中的GDB的集成

Android 的gdbclient 本质是一个脚本：包含了GDB工具链的ENV，共享库配置ENV，file 搜索路径ENV等

Get gdbserver[编辑 | 编辑源代码]

如前面描述gdbserver 运行于目标版，所以需要在AOSP 中找到正确的版本，两个条件CPU ARCH 和 位数，我们手机那就是ARM64了，默认会根据

变量TARGET_ARCH=arm64 编译对应的gdbserver，输出路径类似：

out/target/product/mobeeplus00b_msm8916_64/obj/PACKAGING/target_files_intermediates/mobeeplus00b_msm8916_64-target_files-eng.zhl/SYSTEM/bin/gdbserver64

adb push out/target/product/mobeeplus00b_msm8916_64/obj/PACKAGING/target_files_intermediates/mobeeplus00b_msm8916_64-target_files-eng.zhl/SYSTEM/bin/gdbserver64 /system/bin/

chmod 755 /system/bin/ gdbserver64

此外还可以从AOSP prebuild中找到对应的gdbserver64，如ARM64:android/prebuilts/misc/android-arm64/gdbserver64

function gdbclient() {
  # TODO:
  # 1. Check for ANDROID_SERIAL/multiple devices
  local PROCESS_NAME="n/a"
  local PID=$1
  local PORT=5039
  if [ -z "$PID" ]; then
    echo "Usage: gdbclient <pid|processname> [port number]"
    return -1
  fi
  #local DEVICE=$(adb_get_product_device)
  local DEVICE=$TARGET_PRODUCT     //获得DEVICE，后面用来生成file和共享库搜索路径
 
  if [ -z "$DEVICE" ]; then
    echo "Error: Unable to get device name. Please check if device is connected and ANDROID_SERIAL is set."
    return -2
  fi
 
  if [ -n "$2" ]; then
    PORT=$2
  fi
 
  local ROOT=$(gettop)
  if [ -z "$ROOT" ]; then
    # This is for the situation with downloaded symbols (from the build server)
    # we check if they are available.
    ROOT=`realpath .`
  fi
 
  local OUT_ROOT="$ROOT/out/target/product/$DEVICE"
  local SYMBOLS_DIR="$OUT_ROOT/symbols"
 
  if [ ! -d $SYMBOLS_DIR ]; then
    echo "Error: couldn't find symbols: $SYMBOLS_DIR does not exist or is not a directory."
    return -3
  fi
 
  # let's figure out which executable we are about to debug
 
  # check if user specified a name -> resolve to pid
  if [[ ! "$PID" =~ ^[0-9]+$ ]] ; then
    PROCESS_NAME=$PID
    PID=$(pid --exact $PROCESS_NAME)
    if [ -z "$PID" ]; then
      echo "Error: couldn't resolve pid by process name: $PROCESS_NAME"
      return -4
    fi
  fi
 
  local EXE=`adb shell readlink /proc/$PID/exe | sed s/.$//`
  # TODO: print error in case there is no such pid
  local LOCAL_EXE_PATH=$SYMBOLS_DIR$EXE
 
  if [ ! -f $LOCAL_EXE_PATH ]; then
    echo "Error: unable to find symbols for executable $EXE: file $LOCAL_EXE_PATH does not exist"
    return -5
  fi
 
  local USE64BIT=""
 
  if [[ "$(file $LOCAL_EXE_PATH)" =~ 64-bit ]]; then
    USE64BIT="64"
  fi
 
  local GDB=
  local GDB64=
  local CPU_ABI=`adb shell getprop ro.product.cpu.abilist | sed s/.$//`
  # TODO: we assume these are available via $PATH
  if [[ $CPU_ABI =~ (^|,)arm64 ]]; then
    GDB=arm-linux-androideabi-gdb
    GDB64=aarch64-linux-android-gdb
  elif [[ $CPU_ABI =~ (^|,)arm ]]; then
    GDB=arm-linux-androideabi-gdb
  elif [[ $CPU_ABI =~ (^|,)x86_64 ]]; then
    GDB=x86_64-linux-androideabi-gdb
  elif [[ $CPU_ABI =~ (^|,)x86 ]]; then
    GDB=x86_64-linux-androideabi-gdb
  elif [[ $CPU_ABI =~ (^|,)mips64 ]]; then
    GDB=mipsel-linux-android-gdb
    GDB64=mips64el-linux-android-gdb
  elif [[ $CPU_ABI =~ (^|,)mips ]]; then
    GDB=mipsel-linux-android-gdb                     //此处根据CPU ARCH 和位数，找到对应的GDB 交叉工具链
  else
    echo "Error: unrecognized cpu.abilist: $CPU_ABI"
    return -6
  fi
 
  # TODO: check if tracing process is gdbserver and not some random strace...
  if [ $(adb_get_traced_by $PID) -eq 0 ]; then
    # start gdbserver
    echo "Starting gdbserver..."
    # TODO: check if adb is already listening $PORT
    # to avoid unnecessary calls
    echo ". adb forward for port=$PORT..."
    adb forward tcp:$PORT tcp:$PORT
    echo ". starting gdbserver to attach to pid=$PID..."
    adb shell gdbserver$USE64BIT :$PORT --attach $PID &
    echo ". give it couple of seconds to start..."   //根据传入参数pid,找到gdbserver64需要attch的进程
    sleep 2
    echo ". done"
  else
    echo "It looks like gdbserver is already attached to $PID (process is traced), trying to connect to it using local port=$PORT"
  fi
 
  local OUT_SO_SYMBOLS=$SYMBOLS_DIR/system/lib$USE64BIT
  local OUT_VENDOR_SO_SYMBOLS=$SYMBOLS_DIR/vendor/lib$USE64BIT
  local ART_CMD=""
 
  echo >|"$OUT_ROOT/gdbclient.cmds" "set solib-absolute-prefix $SYMBOLS_DIR"
  echo >>"$OUT_ROOT/gdbclient.cmds" "set solib-search-path $OUT_SO_SYMBOLS:$OUT_SO_SYMBOLS/hw:$OUT_SO_SYMBOLS/ssl/engines:$OUT_SO_SYMBOLS/drm:$OUT_SO_SYMBOLS/egl:$OUT_SO_SYMBOLS/soundfx:$OUT_VENDOR_SO_SYMBOLS:$OUT_VENDOR_SO_SYMBOLS/hw:$OUT_VENDOR_SO_SYMBOLS/egl"   //设置共享库路径
  local DALVIK_GDB_SCRIPT=$ROOT/development/scripts/gdb/dalvik.gdb
  if [ -f $DALVIK_GDB_SCRIPT ]; then
    echo >>"$OUT_ROOT/gdbclient.cmds" "source $DALVIK_GDB_SCRIPT"
    ART_CMD="art-on"
  else
    echo "Warning: couldn't find $DALVIK_GDB_SCRIPT - ART debugging options will not be available"
  fi
  echo >>"$OUT_ROOT/gdbclient.cmds" "target remote :$PORT"
  if [[ $EXE =~ (^|/)(app_process|dalvikvm)(|32|64)$ ]]; then
    echo >> "$OUT_ROOT/gdbclient.cmds" $ART_CMD
  fi
 
  echo >>"$OUT_ROOT/gdbclient.cmds" ""
 
  local WHICH_GDB=$GDB
 
  if [ -n "$USE64BIT" -a -n "$GDB64" ]; then
    WHICH_GDB=$GDB64
  fi
 
  gdbwrapper $WHICH_GDB "$OUT_ROOT/gdbclient.cmds" "$LOCAL_EXE_PATH"
}

connect via wifi[编辑 | 编辑源代码]

在某些情况下，USB无法使用，比如USB模块挂掉以后，无法使用adb/via via usb

这个时候还想使用gdb online的话，可以使用adb/gdb via wifi 保证几点即可：

1:手机和电脑处在通一个局域网，比如手机和电脑均连接上通一个AP

2:gdbserver[64]attch 或启动一个进程时， 显示指定网络ip即可，eg:gdbserver64 192.168.1.101:1234 --attach 2758

3:target remote 192.168.1.101:1234 来连接手机即可， 192.168.1.101 为手机的ip地址

备注[编辑 | 编辑源代码]

1：调试apk pid时，默认会去调试app_process[64]

2：被调试的进程被gdb attch后，默认会处于暂停状态， 比如调试surfaceflinger时，显示系统会出现“冻结”状态，不用紧张...正等待您下达指令

3：gdb也是可以调试内核线程的，比如一个kworker,但是基于内核特点和Hardware Watch Dog的存在，不方便“冻结”内核，gdb调试内核线程前需要较多的配置，

比如disable Hardware Watch Dog等，恰恰这些和ARCH关联性很大，各个ARCH差异较大， 需要下次wiki再做详解
