[编辑] 背景

spirit手机在测试和使用中，出现了严重死机的问题，具体表现
(1). mtp/adb传输大量数据时
(2). 视频录制一段时间后
(3). 使用"A1 SD Bench.apk"测试internal memory(data分区)读写性能
(4). 使用dd进行大量的读写测试

[编辑] 见招拆招

我们在测试过程中，发现这个问题是必现的，当数据读写到一定阶段之后就会发生，而且对比A的手机没有这种问题。因此做了以下的怀疑：
1. emmc的读写问题，在抓取log后发现出现问题的log和之前荣刚做emmc兼容机制出错的信息完全一致，于是先去排除emmc的兼容机制导致了问题，但是在去掉emmc兼容的resize后，发现问题依然存在。
2. ext4文件系统问题，在出现问题的操作中都必须通过文件系统，但是因为多个平台都共用同样的文件系统代码，基本可以排除这种问题，而且如果真的是存在问题，我们也无法解决。同时这种问题可以考虑使用一个没有文件系统的分区做DD测试。
3. ram的问题，这种问题分成两类，一类是ddr的硬件问题，毕竟spirit使用了和A不同的emmc+ram芯片，所以荣刚先从这边入手，排查ddr的datasheet；ram软件机制或者错误的问题，我和启东从这边入手。

从backtrace的调用过程中，可以粗略的看到ext4写一个文件的流程，
每次写文件时，ext4需要通过kernel申请一个page作为buffer，申请page的函数从grab_cache_page_write_begin()开始，就进入了kernel较为核心的memory management部分。
kernel在分配一个page的时候，会调用get_page_from_freelist()来查找kernel认为是空闲的页。kernel会维护一个free 页的链表，每次申请时，kernel就会在这个表中查询，并且分配给申请的用户或者内核进程。
当然get_page_from_freelist这个函数远远没有这么肤浅，可以简单看下它的参数： 

get_page_from_freelist(gfp_t gfp_mask, nodemask_t *nodemask, unsigned int order,
		struct zonelist *zonelist, int high_zoneidx, int alloc_flags,
		struct zone *preferred_zone, int migratetype)

内核的内存管理太深奥，只能简单做个解释：
我们通常理解的内存对于内核来说分成三个区域，就是zone，
ZONE_DMA占了低于16M的内存页框，
ZONE_NORMAL占了16M到896M的内存页框，
ZONE_HIGHMEM在32位的系统中，包含了从896M开始高于896到4G范围的内存页框。
因此kernel在分配的时候，通常会在normal和highmem找空闲的页，这部分的分配也很有讲究，kernel会对每个zone做watermark标记，watermark可以理解成水位线，当某个zone的水位线到临界点时，kernel不会再对这个zone进行分配，转而去找其他高于水位线的zone进行分配；当内存分配很紧张，大家都临界水位线时，kernel就尝试回收一部分页，再进行分配；如果还是不够，就只有尝试降低水位线。
在整个分配过程中，kernel利用伙伴系统算法进行分配，尽可能避免碎片化的产生，但是当申请大块内存相当密集的时候，内存的碎片化还是不可避免。
可以用cat proc/buddyinfo来查看zone的碎片化情况：

图中的数值，是从左方开始，以2的n次方取值，乘以每个页的大小，表示现在zone中可以使用的连续的内存有多少，667表示有667个不连续的页可供分配，它右边表示有560个2*PAGE_SIZE的页可供分配，以此类推。简单说，左边的值越大右边值越小，碎片化越严重。

当然以上这些跟我们的问题关系不大，还要接着往下看。
根据backtrace，最终我们还是会找到kernel认为可以分配的页面，来到这个函数： 

prep_new_page(page, order, gfp_flags)

这个函数会调用 

void kernel_map_pages(struct page *page, int numpages, int enable)
{
    if (enable)
        unpoison_pages(page, numpages);
    else
        poison_pages(page, numpages);
}

kernel_map_pages是这个问题的关键点了，我们在代码中发现，分配和释放页时都会调用这个函数，不同的是，当分配时enable值为1,，释放时enable值为0。
同时，这里有一个冷门又非常关键的知识，就是内核毒剂。接着往下看之前，最好先去了解一下内核毒剂的原理，隆重推介启东的wiki： 

http://10.120.10.100:9002/index.php/Linux_Posion_Introduction

继续看unpoison_pages(page, numpages)和poison_pages(page, numpages)这两个函数： 

static void unpoison_page(struct page *page)
{
    void *addr;

    if (!page_poison(page))
        return;

    addr = kmap_atomic(page);
    check_poison_mem(addr, PAGE_SIZE);
    mark_addr_rdwrite(addr);
    clear_page_poison(page);
    kunmap_atomic(addr);
}

当申请一个页时，内核调用upoison_pages，内核在给你之前，它先要自己检查一遍这个页是不是真的安全，就会调用check_posion_mem()： 

static void check_poison_mem(unsigned char *mem, size_t bytes)
{
    static DEFINE_RATELIMIT_STATE(ratelimit, 5 * HZ, 10);
    unsigned char *start;
    unsigned char *end;

    start = memchr_inv(mem, PAGE_POISON, bytes);
    if (!start)
        return;

    for (end = mem + bytes - 1; end > start; end--) {
        if (*end != PAGE_POISON)
            break;
    }

    if (!__ratelimit(&ratelimit))
        return;
    else if (start == end && single_bit_flip(*start, PAGE_POISON))
        printk(KERN_ERR "pagealloc: single bit error\n");
    else
        printk(KERN_ERR "pagealloc: memory corruption\n");

    print_hex_dump(KERN_ERR, "", DUMP_PREFIX_ADDRESS, 16, 1, start,
            end - start + 1, 1);
    BUG_ON(PANIC_CORRUPTION);
    dump_stack();
}

于是可以看到，我们之前的问题就崩溃在这里，所以是内存中毒了么？做结论前，还要继续看看，不展示memchr_inv的内容，只看看它的注释： 

<span style="color:red">memchr_inv - Find an unmatching character in an area of memory.</span>

是的，它在检查当前页中是否有一个字节不是PAGE_POSION，PAGE_POSION是什么？就是 

#define PAGE_POISON 0xaa

定义在kernel\include\linux\Poison.h中。
所以看起来是，检查整个页是否都是0xaa，如果有一个字节不是，那么就完蛋了。
这和我们通常理解的中毒是不是有点不一样，好吧，为了澄清这个问题，还要再返回去看看posion_pages的函数实现： 

static void poison_page(struct page *page)
{
    void *addr = kmap_atomic(page);

    set_page_poison(page);
    memset(addr, PAGE_POISON, PAGE_SIZE);
    mark_addr_rdonly(addr);
    check_poison_mem(addr, PAGE_SIZE);
    kunmap_atomic(addr);
}

前面说过，每次释放一个页的时候最终都会来到这里，它竟然把被释放的页面全部染成0xaa，然后再做检查，确认页面完全被毒后才会还给kernel放回到可用的页面中去。
那么，终于是一个轮回了，kernel在回收页面时，将被释放的页面全部染毒，然后放回free list中，下次申请时就将free list中的页拿出来用，用之前先检查一遍，发现页面没有被全被染毒，或者说没有被正确释放，kernel就认为内存被污染，直接panic。

[编辑] 僵局

到了这里，我们已经知道了问题的深层次原因，就是内存在kernel不知情的情况下被污染，kernel仍然傻乎乎的拿这块内存来招待客人。
但是，这并不能说明是ddr本身的问题，还是野指针造成的问题。
ddr如果在某个地址上会固定出现问题，有可能造成这样的污染，并且这几乎是无法拯救的。高通的高手也开始指点我们去查找ddr稳定性的问题。
如果是野指针，那么好吧，运行的程序千千万，哪里去茫茫人海中找一个隐藏的野指针。

[编辑] 破局

荣刚贡献了临门一脚，玄机就在开头的那段log中，dff00000地址被固定写入了一个字符串，这个字符串就是DBGC，换成ascii码就是0x44,0x42,0x47,0x43。
通过在代码中查找DBGC这个字符串，我们最终找到了在kernel/fs/pstore/ram_core.c中 

#define PERSISTENT_RAM_SIG (0x43474244) /* DBGC */

以及kernel/fs/pstore/ram.c中 

static ulong mem_address=0x9ff00000;

这个地址换算过来，dff00000正好向后偏移了1G。
那么问题就这在里，A的dts中声明了这块内存留给pstore专用： 

pstore_reserve_mem: pstore_reserve_mem_region@0 { 
          linux,reserve-contiguous-region; 
          linux,reserve-region; 
          linux,remove-completely; 
          reg = <0x0 0x9ff00000 0x0 0x00100000>; 
          label = "pstore_reserve_mem"; 
};

于是kernel知道这块内存已经被使用，不会再对它进行分配，而8939的dts并没有声明这块内存，于是造成了这一切的问题。

[编辑] 解决的方法

还用说么，给8939的dts中加上使用内存的声明就好了。



以上问题算是解决了。留下几个点需要注意
1. 写固定地址的内存，一定要非常小心，来来去去一定要弄明白才能这样干。
2. 内核毒剂是个好东西，它能帮你了解问题本质。
3. 发掘问题，但是不要掉进去出不来，最终很可能还是要回到最初的那句log。
